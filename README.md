## Enhancement Two: Algorithms and Data Structures

In 2025, I took CS 340, Client/Server Development, and developed an Animal Shelter Dashboard using Python, Dash, MongoDB, and interactive data visualization tools. This dashboard was created to help users view and filter animal shelter data based on different rescue categories. The original dashboard included a data table, rescue-type filters, a geolocation map, and visual elements that allowed users to interact with the animal data. The purpose of the application was to make a large dataset easier to search, filter, and understand.

For my ePortfolio, I selected this artifact for Category Two because it gives me an opportunity to demonstrate my skills with algorithms and data structures. The original dashboard was functional, but much of the data processing could be improved. For this enhancement, I focused on how the dashboard processes filtered animal records after a user selects a rescue category. Instead of only displaying filtered results, I enhanced the program so it organizes and analyzes the filtered data using dictionaries, sets, and a single traversal through the dataset. This allows the dashboard to function as more than a database query display because the application tier now performs structured in-memory analysis on the records returned from MongoDB.

The main enhancement I made was adding a helper function that processes the filtered animal records more efficiently. This helper function unwraps the filtered row data into a consistent structure, isolates the string-based fields needed for analysis, and builds reusable data models during one pass through the dataset. For each animal record, the function extracts values such as breed, outcome type, and age, then uses those values to update the proper collection. A dictionary is used to count breeds, another dictionary is used to count outcome types, and a set is used to track unique breeds. The function also calculates the total number of animals and the average age of the animals in the filtered results. To prevent runtime errors, the logic checks for missing, blank, or null values before using them in calculations or collection updates. This makes the enhancement stronger because real datasets often contain incomplete records that must be handled safely.

Specific components of the enhanced dashboard showcase my skills in algorithms and data structures because they show how I can process and organize data beyond basic display functionality. The breed count dictionary stores each breed as a key and the number of matching animals as the value. This allows the program to quickly update breed totals and reuse those totals for the chart and summary statistics. The outcome count dictionary works in a similar way by tracking how often each outcome appears in the filtered records. The unique breed set prevents duplicate breed names and allows the dashboard to quickly determine how many different breeds are represented in the current filter. These structures are appropriate because dictionary and set lookups operate in average O(1) time, which means the program can update counts and check uniqueness efficiently as the dataset grows.

The updated dashboard also uses the processed results to display summary statistics, such as total animals, unique breeds, most common breed, and average age. These statistics make the dashboard more useful because users can quickly understand the filtered data without manually reviewing every row in the table. The chart is also generated from the dictionary-based breed counts, which shows that the data structures are not only created in the background but are actively used to support the dashboard's visual output. By producing the table, chart, and summary statistics from the same processed collection, the enhancement avoids repeating the same work in multiple parts of the application.

This enhancement supports the course outcome related to applying algorithms and data structures to solve problems. The enhanced function processes the filtered records in O(n) time because each animal record is visited once. In this case, n represents the number of filtered animal records returned for the selected rescue category. A less efficient design could scan the same dataset multiple times to calculate total animals, breed counts, outcome totals, unique breeds, and average age separately. That approach would still be linear for each individual scan, but the repeated passes would create unnecessary overhead, such as O(5n), and a nested design that compares records against other records could degrade toward O(n²). My single-pass approach keeps the work proportional to the dataset size while building several outputs at the same time. Because dictionary and set operations are average O(1), the dashboard avoids performance degradation that could occur if each lookup required a full search through a list.

The helper function also improves the architecture of the application by separating data-wrangling logic from the dashboard's visual layout. Instead of placing the analysis logic directly inside the table, chart, or callback display code, the function acts as a clean processing layer that receives filtered records and returns reusable summary data. This decoupled design makes the program easier to test because the helper function can be checked independently from the user interface. It also makes the logic reusable because the same processed results can support different visualizations, such as a chart, summary card, or future dashboard component, without rewriting the algorithm each time.

I believe I met the course outcomes I planned to meet with this enhancement. In Module One, I planned to use the Animal Shelter Dashboard to demonstrate my ability to work with algorithms and data structures. This enhancement meets that goal because it uses dictionaries, sets, and list traversal to improve how the dashboard processes animal data. The enhancement also supports the outcome related to designing and evaluating computing solutions because I improved the dashboard in a way that makes the application more efficient, easier to maintain, and more useful to the user. The result is a stronger Category Two artifact because the focus remains on algorithmic processing, data structure selection, and performance rather than only on user interface design or database connectivity.

This enhancement helped me better understand how important data structures are when working with large datasets. Before this enhancement, I mostly viewed the dashboard as a database and visualization project. After working on it, I can better see how algorithms and data structures affect the way information is processed and displayed. One challenge I faced was making sure the enhancement clearly showed algorithmic thinking instead of only adding another dashboard feature. By focusing on dictionary counts, set-based unique values, safe handling of incomplete records, and single-pass processing, I was able to strengthen the artifact for Category Two. Overall, this enhancement helped me improve my ability to organize data efficiently and explain the reasoning behind my design choices.

### Database Enhancement Code Snippet

```python
def read(self, data, projection=None):
    # If no query is provided, return all records
    if data is None or data == {}:
        query = {}

    # Verifies inputted data is correct type
    elif not isinstance(data, dict):
        raise Exception("Invalid data type")

    else:
        query = {}

        for key, value in data.items():
            # Verifies that key and/or value is not null
            if key is None or value is None:
                return []

            # Verifies key is correct type
            if not isinstance(key, str):
                return []

            # Verifies key is not empty
            if key.isspace() or key == "":
                return []

            # Verifies value is correct type and normalizes value
            if isinstance(value, str):
                cleaned = value.strip()

                if cleaned == "":
                    return []

                query[key] = cleaned
            else:
                query[key] = value

    # Default projection removes MongoDB ObjectId
    if projection is None:
        projection = {"_id": 0}

    # Returns a list with matching results of query
    results = list(self.database.animals.find(query, projection))
    return results
```
